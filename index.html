MODULE 01

Core Concepts

Everything you need to know before writing a single line of code.

Non-Technical Concepts

🤖

What is an AI Agent?

An agent is an AI system that can perceive inputs, reason about them, and take actions — not just respond. Unlike a chatbot that answers questions, an agent decides what to do next to reach a goal. Think of it like an employee who doesn't just answer questions but actually does the work.

🧠

What is an LLM?

A Large Language Model (LLM) is the "brain" powering AI agents. It's a neural network trained on vast text that can understand and generate language. In FinBot, the LLM (GPT-4o) is what understands "Should I invest in index funds?" and reasons through an answer.

🔧

What are Tools?

Tools are functions that agents call to interact with the real world. Without tools, agents are stuck in text. With tools, they can fetch stock prices, read your bank transactions, send emails, or query databases. FinBot will have tools like get_expense_data() and calculate_budget().

🔄

The Agent Loop

Agents work in a cycle: Think → Act → Observe → Repeat. The LLM decides what to do, calls a tool (Act), gets the result (Observe), and decides what to do next. This loop continues until the task is complete. This is called ReAct (Reasoning + Acting).

💡
Finance analogy: Think of a human financial advisor. They don't just answer your question — they look up your account history (tool use), analyze your situation (reasoning), consult their knowledge (LLM), and give recommendations (output). ADK lets you replicate this entire workflow programmatically.

Technical Concepts

What is Google ADK?

Google's Agent Development Kit (ADK) is an open-source Python framework for building, testing, and deploying AI agents. It provides:

  • A structured way to define agents and their behaviors
  • Built-in tool management and function calling
  • Session and memory management
  • Multi-agent orchestration (agents calling other agents)
  • Deployment utilities for Google Cloud

Common Errors — Module 1

Error "Agent" vs "Chatbot" confusion

Symptom: You build something that just answers questions but never calls tools.
Fix: Ask yourself: "Is the agent doing something, or just saying something?" If only saying — add a tool.

Error Overcomplicating the first agent

Symptom: Trying to build the full multi-agent system before a single agent works.
Fix: Follow the modules in order. Get a single agent with one tool working before adding sub-agents or memory.

Error Choosing the wrong framework

Symptom: Mixing ADK concepts with LangChain docs and getting confused.
Fix: Stick to one framework (ADK) for this course. The concepts transfer, but the APIs differ. ADK docs: google.github.io/adk-docs

Your First Agent

ADK Core Architecture

Every ADK agent has three parts: the model (the LLM brain), the instruction (system prompt), and optionally tools. We'll add tools in Module 4.

Colab Cell / agents/finbot_agent.py
from google.adk.agents import Agent
from google.adk.models.lite_llm import LiteLlm
import os

# ──────────────────────────────────────────────
# Step 1: Configure the LLM (OpenAI GPT-4o)
# ADK uses LiteLLM as a universal LLM adapter
# ──────────────────────────────────────────────

openai_model = LiteLlm(
    model="openai/gpt-4o",              # Model identifier

)

# ──────────────────────────────────────────────
# Step 2: Define the Agent
# ──────────────────────────────────────────────

    name="finbot",
    model=openai_model,
    description="A personal finance advisor that helps users manage money",
    instruction="""You are FinBot, a friendly and knowledgeable personal 
    finance advisor. You help users:
    - Understand their spending patterns
    - Create and stick to budgets
    - Make informed investment decisions
    - Plan for financial goals
    
    Always be encouraging, never judgmental about past spending.
    Provide actionable advice. Ask clarifying questions when needed.
    You MUST disclose you are an AI and recommend professional 
    advice for major financial decisions."""
)

print(f"✅ FinBot agent created: {finbot.name}")

Running the Agent

Colab Cell / main.py
from google.adk.runners import InMemoryRunner
from google.adk.sessions import InMemorySessionService
from google.genai import types

# Import our agent
from agents.finbot_agent import finbot

# ──────────────────────────────────────────────
# Step 3: Create a session (tracks conversation)
# ──────────────────────────────────────────────

session_service = InMemorySessionService()
session = session_service.create_session(
    app_name="finbot_app",
    user_id="user_001",
    session_id="session_001"
)

# ──────────────────────────────────────────────
# Step 4: Create a runner (executes the agent)
# ──────────────────────────────────────────────

runner = InMemoryRunner(
    agent=finbot,
    app_name="finbot_app",
    session_service=session_service
)

# ──────────────────────────────────────────────
# Step 5: Send a message!
# ──────────────────────────────────────────────

def chat(user_message: str):
    """Helper function to send a message to FinBot"""
    response = runner.run(
        user_id="user_001",
        session_id="session_001",
        new_message=types.Content(
            role="user",
            parts=[types.Part(text=user_message)]
        )
    )
    # Extract the text response
    for event in response:
        if event.is_final_response():
            print(f"FinBot: {event.content.parts[0].text}\n")
            return

# Test it!
chat("Hi! I earn $5000/month and spend about $4200. Am I saving enough?")
🔍
What just happened? The user message went to the Runner → Runner passed it to the Agent → Agent sent it to GPT-4o with the system instruction → GPT-4o responded → ADK returned the response through events. The Session stored the conversation history so FinBot remembers context.

🧪 Quick Check — What is the role of the instruction parameter in an ADK Agent?

It defines which Python tools the agent can call
It sets the agent's name for logging purposes
It is the system prompt — it defines the agent's persona, behavior rules, and task focus
It specifies which LLM model version to use
✅ Correct! The instruction is sent to the LLM as its system prompt before every conversation. It's where you define personality, rules, constraints, and task focus. Treat it like a job description for the agent.

🧪 Quick Check — What does event.is_final_response() check for?

Whether the agent encountered an error
Whether the event contains the agent's final text response to the user (vs intermediate tool calls)
Whether the session has ended
Whether all tool calls have completed
✅ Correct! ADK returns a stream of events — some are tool call events, some are intermediate thoughts, and one is the final text response. is_final_response() filters for that last one, which is what you display to the user.

Common Errors — Module 3

Error Agent creates but runner hangs / no output

Cause: The runner.run() call returns a generator — you must iterate over it.
Fix: Make sure you have for event in response: loop. Without iterating, nothing executes. The events are lazy.

Error Session already exists

Cause: Running create_session() twice with the same session ID in Colab (re-running a cell).
Fix: Use get_or_create_session() instead, or add a try/except around create_session(). Alternatively use a unique session ID each run: session_id=str(uuid.uuid4()).

Error Agent responds in wrong language or ignores instructions

Cause: Weak or ambiguous instruction prompt. LLMs need clear, specific guidance.
Fix: Be more explicit: "You MUST always respond in English." "You MUST always ask the user's income before giving budget advice." The word MUST carries weight.

Give FinBot superpowers — the ability to fetch, calculate, and analyze real data.

🏗️

How Tool Calling Works

When you define a Python function as a tool, ADK tells the LLM it exists (via the function's docstring and type hints). The LLM decides when to call it. This is called function calling or tool use.

User: "How much did I spend on food?"
↓
LLM decides: I need to call
get_expenses(category="food")
↓
ADK calls the Python function
Returns: $487.50
↓
LLM synthesizes: "You spent $487.50 on food..."

import json
from datetime import datetime, timedelta
from typing import Optional

SAMPLE_EXPENSES = [
    {"date": "2024-01-05", "amount": 85.00, "category": "food", "desc": "Groceries"},
    {"date": "2024-01-08", "amount": 45.00, "category": "food", "desc": "Restaurant"},
    {"date": "2024-01-10", "amount": 1200, "category": "housing", "desc": "Rent"},
    {"date": "2024-01-12", "amount": 65.00, "category": "transport", "desc": "Gas"},
    {"date": "2024-01-15", "amount": 120, "category": "utilities", "desc": "Electricity"},
    {"date": "2024-01-20", "amount": 200, "category": "entertainment", "desc": "Subscriptions"},
]

def get_monthly_expenses(
    category: Optional[str] = None,
    month: Optional[str] = None
) -> dict:
    """
    Retrieve expense records for a given month and/or category.
    
    Args:
        category: Filter by category (food, housing, transport, 
                  utilities, entertainment). None returns all.
        month: Month in YYYY-MM format. None returns current month.
    
    Returns:
        dict with 'total', 'count', and 'items' list
    """
    expenses = SAMPLE_EXPENSES
    
    # Filter by category if specified
    if category:
        expenses = [e for e in expenses 
                   if e["category"].lower() == category.lower()]
    
    total = sum(e["amount"] for e in expenses)
    
    return {
        "total": round(total, 2),
        "count": len(expenses),
        "items": expenses,
        "category": category or "all"
    }

def calculate_budget_health(
    monthly_income: float,
    monthly_expenses: float
) -> dict:
    """
    Calculate budget health metrics and the 50/30/20 rule score.
    
    Args:
        monthly_income: Total monthly income in dollars
        monthly_expenses: Total monthly spending in dollars
    
    Returns:
        dict with savings_rate, surplus, health_score, and recommendations
    """
    savings = monthly_income - monthly_expenses
    savings_rate = (savings / monthly_income) * 100 if monthly_income > 0 else 0
    
    # Determine health score
    if savings_rate >= 20:
        health = "excellent"
    elif savings_rate >= 10:
        health = "good"
    elif savings_rate >= 0:
        health = "needs_improvement"
    else:
        health = "critical"
    
    return {
        "monthly_income": monthly_income,
        "monthly_expenses": monthly_expenses,
        "monthly_savings": round(savings, 2),
        "savings_rate_pct": round(savings_rate, 1),
        "health_status": health,
        "fifty_thirty_twenty": {
            "needs_50pct": round(monthly_income * 0.5, 2),
            "wants_30pct": round(monthly_income * 0.3, 2),
            "savings_20pct": round(monthly_income * 0.2, 2),
        }
    }

🧪 Quick Check — How does the LLM know when to call a tool?

You must hardcode an if/else in your Python code to trigger tool calls
The LLM reads the function's docstring and type hints, then decides autonomously when the tool is needed
ADK automatically calls all tools for every user message
The user must type a special command like /tool get_expenses
✅ Correct! This is the power of function calling. You don't write routing logic — the LLM reads the docstring and decides "this user is asking about expenses, so I should call get_monthly_expenses()." Clear docstrings = better tool selection.

🧪 Quick Check — What happens if a tool function raises a Python exception?

The entire agent crashes and the session is lost
The exception is silently ignored and the agent continues
ADK passes the error back to the LLM as a tool result, which the LLM can reason about and handle gracefully
The agent automatically retries the tool call 3 times
✅ Correct! ADK is resilient — it catches tool exceptions and surfaces them to the LLM as error results. A well-prompted agent will handle this gracefully (e.g., "I couldn't fetch your expenses right now. Could you tell me your spending manually?"). Always test your tools in isolation first though.

Common Errors — Module 4

Error Agent never calls the tool

Cause: Vague docstring, missing type hints, or tool not added to the tools=[] list.
Fix: (1) Check the tool is in tools=[]. (2) Add a line to instruction: "When asked about expenses, ALWAYS call get_monthly_expenses() first." Explicit beats implicit.

Error Tool called with wrong argument types

Cause: LLM passes "85" (string) instead of 85.0 (float) for a numeric param.
Fix: Add input coercion at the top of your tool: monthly_income = float(monthly_income). Defensive typing is essential for LLM-called functions.

Error TypeError: get_monthly_expenses() got unexpected keyword argument

Cause: The LLM hallucinated a parameter name that doesn't exist in the function signature.
Fix: Add **kwargs to absorb unexpected args, or make your docstring parameter names very explicit and unambiguous. Example: rename cat to category.

MODULE 05

Memory & State

Make FinBot remember your financial profile across conversations.

The Three Types of Memory in ADK

In-Session Memory

What: The current conversation history. Automatically managed by ADK's Session object.
FinBot use: Remembers "you said your income is $5000" during the current chat.

Persistent Memory

What: Data stored between sessions (in a DB or file). Requires explicit saving.
FinBot use: Stores your financial profile — income, goals, risk tolerance — permanently.

External Memory (RAG)

What: A knowledge base the agent can search. Built with vector databases.
FinBot use: Search through financial documents, past advice, or articles.

Colab Cell Persistent User Profile
import json
from pathlib import Path

# ── Simple file-based persistent memory ──────────────────
# In production: use a database (Firestore, PostgreSQL, etc.)

PROFILES_FILE = Path("data/user_profiles.json")

def save_user_profile(
    user_id: str,
    monthly_income: float,
    financial_goals: str,
    risk_tolerance: str
) -> dict:
    """
    Save or update the user's financial profile for future sessions.
    
    Args:
        user_id: Unique user identifier
        monthly_income: User's monthly take-home income
        financial_goals: Main goals (e.g., 'buy house in 5 years')
        risk_tolerance: Investment risk level (low/medium/high)
    
    Returns:
        Confirmation dict with saved profile data
    """
    profiles = {}
    if PROFILES_FILE.exists():
        profiles = json.loads(PROFILES_FILE.read_text())
    
    profiles[user_id] = {
        "monthly_income": monthly_income,
        "financial_goals": financial_goals,
        "risk_tolerance": risk_tolerance,
        "updated_at": str(datetime.now())
    }
    
    PROFILES_FILE.parent.mkdir(exist_ok=True)
    PROFILES_FILE.write_text(json.dumps(profiles, indent=2))
    
    return {"status": "saved", "profile": profiles[user_id]}

def get_user_profile(user_id: str) -> dict:
    """
    Retrieve the user's saved financial profile.
    
    Args:
        user_id: Unique user identifier
    
    Returns:
        User profile dict, or empty dict if no profile found
    """
    if not PROFILES_FILE.exists():
        return {"status": "no_profile", "message": "No profile found"}
    
    profiles = json.loads(PROFILES_FILE.read_text())
    profile = profiles.get(user_id, {})
    
    return profile if profile else {"status": "no_profile"}

def search_financial_knowledge(
    query: str,
    n_results: Optional[int] = 2
) -> dict:
    """
    Search the financial knowledge base for relevant advice and information.
    Use this tool when the user asks general financial questions like:
    - How to build an emergency fund
    - How to pay off debt (avalanche vs snowball)
    - Index funds vs active funds
    - Budgeting rules (50/30/20)
    - Retirement account strategies
    
    Args:
        query: The user's financial question or topic to search for
        n_results: Number of relevant document chunks to return (default: 2)
    
    Returns:
        dict with 'results' list of relevant text chunks and their sources
    """
    results = collection.query(
        query_texts=[query],
        n_results=n_results
    )
    
    formatted = []
    for i, doc in enumerate(results["documents"][0]):
        formatted.append({
            "rank": i + 1,
            "content": doc,
            "source": results["metadatas"][0][i]["source"],
            "relevance_distance": round(results["distances"][0][i], 3)
        })
    
    return {
        "query": query,
        "results": formatted,
        "total_found": len(formatted)
    }

RAG makes the LLM smarter at reasoning

🧪 Quick Check — In ChromaDB, what are "embeddings"?

Compressed versions of documents to save storage space
Encryption keys used to secure the vector database
Numerical vectors (arrays of floats) that represent the semantic meaning of text, enabling similarity search
Python objects that wrap document text for faster retrieval
✅ Correct! An embedding model converts text into a vector (e.g., 384 numbers). Similar texts produce similar vectors. ChromaDB stores these vectors and finds the nearest ones to your query vector — that's semantic search. "emergency fund savings" and "how much cash to keep" get similar vectors even though they share no words.

Common Errors — Module 5

Error Collection already exists on re-run

Cause: Running the notebook cell twice tries to create the same ChromaDB collection.
Fix: Use a try/except as shown in the code above — try get_collection() first, create only if it fails. Or call chroma_client.delete_collection("finance_knowledge") before recreating.

Error RAG returns irrelevant results

Cause: Query is too short or vague, or chunks are too large.
Fix: Check relevance_distance — values above 1.5 are usually irrelevant. Add a filter: if result["relevance_distance"] < 1.2. Also try chunking documents into smaller 200-400 character segments.

Error Agent ignores the RAG tool and answers from training data

Cause: The LLM thinks it already knows the answer and skips the tool.
Fix: Strengthen the instruction: "For ALL general finance questions, you MUST call search_financial_knowledge() before responding. Do not answer from memory alone."

Error sentence-transformers takes 5+ minutes to download on Colab

Cause: The model weights (~90MB) download on first use.
Fix: This is normal on first run. On subsequent runs it uses the cache. Alternatively, use OpenAI embeddings (paid, faster) by replacing SentenceTransformerEmbeddingFunction with OpenAIEmbeddingFunction.

def get_portfolio_allocation(
    risk_tolerance: str,
    investment_horizon_years: int,
    monthly_investment_amount: float
) -> dict:
    """
    Calculate a recommended portfolio allocation based on risk profile.
    
    Args:
        risk_tolerance: User's risk level — "low", "medium", or "high"
        investment_horizon_years: How many years until they need the money
        monthly_investment_amount: How much they can invest each month
    
    Returns:
        dict with recommended asset allocation percentages and projected growth
    """
    allocations = {
        "low": {
            "us_total_market_index": 40,
            "international_index": 20,
            "bond_index": 35,
            "cash_money_market": 5,
            "expected_annual_return_pct": 5.5
        },
        "medium": {
            "us_total_market_index": 55,
            "international_index": 25,
            "bond_index": 15,
            "cash_money_market": 5,
            "expected_annual_return_pct": 7.5
        },
        "high": {
            "us_total_market_index": 65,
            "international_index": 25,
            "bond_index": 5,
            "real_estate_reits": 5,
            "expected_annual_return_pct": 9.0
        }
    }
    
    alloc = allocations.get(risk_tolerance.lower(), allocations["medium"])
    rate = alloc["expected_annual_return_pct"] / 100
    monthly_rate = rate / 12
    n_months = investment_horizon_years * 12
    
    # Future Value of monthly contributions (compound interest formula)
    if monthly_rate > 0:
        future_value = monthly_investment_amount * (
            (((1 + monthly_rate) ** n_months) - 1) / monthly_rate
        )
    else:
        future_value = monthly_investment_amount * n_months
    
    total_contributed = monthly_investment_amount * n_months
    
    return {
        "risk_profile": risk_tolerance,
        "investment_horizon_years": investment_horizon_years,
        "monthly_investment": monthly_investment_amount,
        "allocation": alloc,
        "projected_value": round(future_value, 2),
        "total_contributed": round(total_contributed, 2),
        "projected_gains": round(future_value - total_contributed, 2),
        "disclaimer": "Projections assume constant returns. Actual returns vary."
    }

# ── Goal tracking tools ────────────────────────────────────

def create_financial_goal(
    user_id: str,
    goal_name: str,
    target_amount: float,
    target_date: str,
    current_saved: Optional[float] = 0.0
) -> dict:
    """
    Create and save a new financial goal for the user.
    
    Args:
        user_id: Unique user identifier
        goal_name: Descriptive goal name (e.g., "House down payment", "Emergency fund")
        target_amount: Total amount needed in dollars
        target_date: Target completion date in YYYY-MM-DD format
        current_saved: Amount already saved toward this goal (default: 0)
    
    Returns:
        dict with goal details and required monthly savings calculation
    """
    goals = {}
    if GOALS_FILE.exists():
        goals = json.loads(GOALS_FILE.read_text())
    
    if user_id not in goals:
        goals[user_id] = []
    
    # Calculate months remaining
    target_dt = datetime.strptime(target_date, "%Y-%m-%d")
    months_remaining = max(1, (target_dt.year - datetime.now().year) * 12 + 
                          (target_dt.month - datetime.now().month))
    
    amount_remaining = target_amount - current_saved
    monthly_required = round(amount_remaining / months_remaining, 2)
    progress_pct = round((current_saved / target_amount) * 100, 1) if target_amount > 0 else 0
    
    goal = {
        "id": f"goal_{len(goals[user_id]) + 1}",
        "name": goal_name,
        "target_amount": target_amount,
        "current_saved": current_saved,
        "target_date": target_date,
        "months_remaining": months_remaining,
        "monthly_required": monthly_required,
        "progress_pct": progress_pct,
        "created_at": str(datetime.now().date()),
        "status": "on_track"
    }
    
    goals[user_id].append(goal)
    GOALS_FILE.parent.mkdir(exist_ok=True)
    GOALS_FILE.write_text(json.dumps(goals, indent=2))
    
    return goal

def get_all_goals(user_id: str) -> dict:
    """
    Retrieve all financial goals for a user with progress tracking.
    
    Args:
        user_id: Unique user identifier
    
    Returns:
        dict with goals list, total targets, and a summary
    """
    if not GOALS_FILE.exists():
        return {"user_id": user_id, "goals": [], "total_goals": 0}
    
    goals = json.loads(GOALS_FILE.read_text())
    user_goals = goals.get(user_id, [])
    
    total_target = sum(g["target_amount"] for g in user_goals)
    total_saved = sum(g["current_saved"] for g in user_goals)
    
    return {
        "user_id": user_id,
        "goals": user_goals,
        "total_goals": len(user_goals),
        "total_target_amount": total_target,
        "total_saved_so_far": total_saved,
        "overall_progress_pct": round((total_saved / total_target) * 100, 1) if total_target > 0 else 0
    }

def update_goal_progress(
    user_id: str,
    goal_id: str,
    amount_added: float
) -> dict:
    """
    Update progress on an existing financial goal by adding saved amount.
    
    Args:
        user_id: Unique user identifier
        goal_id: The goal's ID (e.g., "goal_1")
        amount_added: New amount saved to add to current_saved
    
    Returns:
        Updated goal dict with new progress percentage and status
    """
    if not GOALS_FILE.exists():
        return {"error": "No goals found. Create a goal first."}
    
    goals = json.loads(GOALS_FILE.read_text())
    user_goals = goals.get(user_id, [])
    
    for goal in user_goals:
        if goal["id"] == goal_id:
            goal["current_saved"] += amount_added
            goal["progress_pct"] = round(
                (goal["current_saved"] / goal["target_amount"]) * 100, 1
            )
            # Recalculate monthly requirement
            target_dt = datetime.strptime(goal["target_date"], "%Y-%m-%d")
            months_left = max(1, (target_dt.year - datetime.now().year) * 12 +
                             (target_dt.month - datetime.now().month))
            remaining = goal["target_amount"] - goal["current_saved"]
            goal["monthly_required"] = round(max(0, remaining / months_left), 2)
            goal["months_remaining"] = months_left
            goal["status"] = "completed" if goal["progress_pct"] >= 100 else "on_track"
            
            goals[user_id] = user_goals
            GOALS_FILE.write_text(json.dumps(goals, indent=2))
            return goal
    
    return {"error": f"Goal {goal_id} not found for user {user_id}"}

Common Errors — Module 6

Error Orchestrator always goes to the same sub-agent

Cause: One sub-agent's description is much broader than others, so it matches everything.
Fix: Make descriptions mutually exclusive and specific. Add negative examples: "Do NOT use this agent for budget or goal questions."

Error Circular delegation (agents calling each other)

Cause: Sub-agent A is given Sub-agent B as a tool, and B has A as a tool.
Fix: Enforce a strict hierarchy — only the orchestrator delegates. Sub-agents should have no other agents in their tools list; only Python functions.

Error Sub-agent response lost / orchestrator ignores it

Cause: The orchestrator's instruction doesn't tell it to use the sub-agent's response.
Fix: Add to orchestrator instruction: "Always incorporate the sub-agent's full response into your final answer. Do not summarize or discard sub-agent outputs."

Common Errors — Module 7

Error HuggingFace model returns 503 Model loading

Cause: Free HuggingFace Inference API cold-starts models. The first request takes 20-60 seconds while the model loads.
Fix: Wait and retry. Add a retry wrapper: for i in range(3): try: ... except: time.sleep(30). For production, use a dedicated Inference Endpoint (paid) which stays warm.

Error Open-source model ignores tools and never calls them

Cause: Many open-source models (especially smaller ones) aren't well-trained on function calling.
Fix: Use models explicitly fine-tuned for tool use: mistralai/Mistral-7B-Instruct-v0.3, meta-llama/Meta-Llama-3.1-8B-Instruct. Add explicit tool call examples in the instruction prompt.

Error RateLimitError on HuggingFace free tier

Cause: Free API has rate limits (~1000 requests/day, ~10 requests/minute).
Fix: Add time.sleep(2) between calls, or upgrade to HuggingFace Pro. For testing, use a small local model via Ollama which has no rate limits.

MODULE 08

Guardrails & Safety

Build a responsible finance AI. Callbacks, output validation, and compliance.

🚨
This module is critical for finance. An AI giving wrong financial advice can cause real harm. Guardrails aren't optional — they're the difference between a demo and a responsible product. In many jurisdictions, automated financial advice has legal requirements.

ADK Callbacks

ADK provides callback hooks that intercept agent behavior at key points. These are the foundation of guardrails:

Colab Cell / agents/guardrails.py
from google.adk.agents.callback_context import CallbackContext
from google.adk.models import LlmResponse, LlmRequest
from google.genai import types
from typing import Optional
import re

# ── Guardrail 1: Block harmful financial advice ───────────

DANGEROUS_PATTERNS = [
    r"guarantee.*return",
    r"risk.free.*invest",
    r"insider.*tip",
    r"100%.* profit",
    r"get rich quick"
]

def validate_finance_output(
    callback_context: CallbackContext,
    llm_response: LlmResponse
) -> Optional[LlmResponse]:
    """
    Callback that runs AFTER the LLM generates a response.
    Scans for dangerous financial claims and blocks them.
    
    Return None to let the response pass through.
    Return a modified LlmResponse to override the output.
    """
    
    if not llm_response.content or not llm_response.content.parts:
        return None
    
    response_text = llm_response.content.parts[0].text.lower()
    
    for pattern in DANGEROUS_PATTERNS:
        if re.search(pattern, response_text, re.IGNORECASE):
            # Override the response with a safe fallback
            safe_response = types.Content(
                role="model",
                parts=[types.Part(text="""I noticed my response may have 
                contained misleading financial language. I want to be 
                clear: no investment is risk-free, and I cannot 
                guarantee returns. Please consult a certified 
                financial advisor for personalized advice. 
                
                I am an AI assistant and my suggestions are 
                informational only.""")]
            )
            return LlmResponse(content=safe_response)
    
    return None  # Response passes through unchanged

# ── Guardrail 2: Always add disclaimer to investment advice ─

def add_disclaimer_to_investments(
    callback_context: CallbackContext,
    llm_response: LlmResponse
) -> Optional[LlmResponse]:
    """Append a legal disclaimer to any investment advice."""
    
    if not llm_response.content:
        return None
    
    text = llm_response.content.parts[0].text
    invest_keywords = ["invest", "stock", "portfolio", "fund", "return"]
    
    if any(kw in text.lower() for kw in invest_keywords):
        disclaimer = "\n\n---\n⚠️ *This is informational only and not financial advice. Consult a licensed advisor.*"
        modified_text = text + disclaimer
        return LlmResponse(
            content=types.Content(
                role="model",
                parts=[types.Part(text=modified_text)]
            )
        )
    return None

🧪 Quick Check — What is the difference between before_model_callback and after_model_callback?

before_model runs before the LLM sees the input (e.g., block bad queries); after_model runs after the LLM responds (e.g., scan output for dangerous claims)
They are identical — ADK runs both in parallel as a safety check
before_model handles tool calls; after_model handles text responses
before_model is for logging only; after_model can modify responses
✅ Correct! before_model_callback intercepts the request before the LLM — use it to block inappropriate questions, inject context, or enforce topic boundaries. after_model_callback intercepts the response — use it to scan for harmful content, add disclaimers, or audit outputs. Both can return a replacement to override behavior.

🧪 Quick Check — Why is regex-based guardrail filtering not sufficient on its own?

Regex is too slow for production use
Python regex doesn't work with Unicode characters in financial text
LLMs can express harmful ideas without using the exact trigger words — semantic understanding is also needed
Regex patterns are too expensive to maintain for a large pattern library
✅ Correct! A model could say "this strategy has historically generated consistent positive returns with minimal downside" — semantically dangerous but matching no regex pattern. For production, layer regex with an LLM-based safety check (e.g., a separate GPT-4o-mini call that classifies the output as safe/unsafe).

Common Errors — Module 8

Error Callback fires but response isn't replaced

Cause: Returning None from a callback means "pass through unchanged." Accidentally returning None when you meant to replace.
Fix: Add a print statement in your callback to confirm it's firing. Check that you're returning a proper LlmResponse object, not just a string.

Error Guardrail blocks legitimate advice

Cause: Regex patterns are too aggressive (e.g., blocking any sentence with "return").
Fix: Test patterns with re.search(pattern, test_text) before deploying. Use word boundaries: r"\bguarantee\b.*\breturn\b" instead of r"guarantee.*return".

Error Disclaimer appended twice

Cause: Callback fires on both the sub-agent response and the orchestrator's synthesis.
Fix: Add a check: if "⚠️" in text: return None — skip if disclaimer already present. Or only attach callbacks to the orchestrator, not sub-agents.

Step 1: Wrap FinBot in FastAPI

File api/server.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from google.adk.runners import InMemoryRunner
from google.adk.sessions import InMemorySessionService
from google.genai import types
from agents.finbot_agent import finbot_safe
import uvicorn

app = FastAPI(title="FinBot API", version="1.0.0")

# Session service (use Redis/Firestore in production)
session_service = InMemorySessionService()
runner = InMemoryRunner(
    agent=finbot_safe,
    app_name="finbot",
    session_service=session_service
)

class ChatRequest(BaseModel):
    user_id: str
    session_id: str
    message: str

class ChatResponse(BaseModel):
    response: str
    agent_name: str

@app.post("/chat", response_model=ChatResponse)
async def chat(request: ChatRequest):
    """Send a message to FinBot and get a response"""
    
    # Ensure session exists
    try:
        session_service.get_session(
            app_name="finbot",
            user_id=request.user_id,
            session_id=request.session_id
        )
    except:
        session_service.create_session(
            app_name="finbot",
            user_id=request.user_id,
            session_id=request.session_id
        )
    
    # Run the agent
    response_text = ""
    agent_used = "finbot"
    
    events = runner.run(
        user_id=request.user_id,
        session_id=request.session_id,
        new_message=types.Content(
            role="user",
            parts=[types.Part(text=request.message)]
        )
    )
    
    for event in events:
        if event.is_final_response():
            response_text = event.content.parts[0].text
            agent_used = event.author
    
    return ChatResponse(response=response_text, agent_name=agent_used)

@app.get("/health")
def health():
    return {"status": "healthy", "agent": "finbot_safe"}

if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=8000)

🧪 Quick Check — Why is Cloud Run preferable to a regular VM for FinBot?

Cloud Run is faster because it uses Google's internal network
Cloud Run scales to zero when idle (no cost) and auto-scales under load — you only pay for actual request time
Cloud Run provides built-in authentication that VMs lack
Cloud Run automatically updates your container when new code is pushed
✅ Correct! For an early-stage FinBot with unpredictable traffic, a VM running 24/7 wastes money during quiet periods. Cloud Run bills per 100ms of actual execution time and cold-starts in ~2 seconds. Perfect for APIs with variable traffic.

🧪 Quick Check — What must change if you use InMemorySessionService in production on Cloud Run?

Nothing — InMemorySessionService works fine in Cloud Run
You must switch to VertexAiRunner specifically
You must replace it with a persistent session service (Firestore, Redis) because Cloud Run instances restart, losing all in-memory data
InMemorySessionService automatically syncs to Google Cloud Storage
✅ Correct! Cloud Run is stateless — each instance starts fresh. In-memory data (sessions, conversation history) is lost on restart or scale-down. For production, store sessions in Firestore (serverless, scales automatically) or Redis (faster, great for session data). Use the session_id as the Firestore document ID.

Common Errors — Module 9

Error gcloud: command not found

Cause: Google Cloud SDK not installed or not in PATH.
Fix: Install from cloud.google.com/sdk/docs/install. On Colab: !curl https://sdk.cloud.google.com | bash, then restart runtime. Verify: gcloud --version.

Error Container builds but crashes on Cloud Run startup

Cause: Missing environment variables, wrong port, or import errors not caught locally.
Fix: Check Cloud Run logs in the GCP console. The most common cause is a missing env var — use --set-env-vars for all keys. Test locally first: docker run -p 8080:8080 -e OPENAI_API_KEY=sk-xxx your-image.

Error API responds but FinBot has no memory between requests

Cause: Each API call creates a new runner/session in InMemoryRunner, losing history.
Fix: Use a consistent session_id per user (store it client-side or in a cookie). The session service must be persistent (Firestore/Redis). The session_id is the thread that holds conversation history.

Error CORS error when calling API from a browser frontend

Cause: FastAPI doesn't allow cross-origin requests by default.
Fix: Add CORS middleware: from fastapi.middleware.cors import CORSMiddleware and app.add_middleware(CORSMiddleware, allow_origins=["*"]). In production, replace "*" with your specific frontend domain.
